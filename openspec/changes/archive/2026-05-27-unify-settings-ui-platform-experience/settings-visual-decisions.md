## 设置 UI 视觉决策记录

本文件记录 `unify-settings-ui-platform-experience` 讨论中已经确认的视觉方向，作为后续设置系统整改的依据。这里描述的是设计和架构约束，不代表当前代码已经全部完成。

## 总原则

MemoFlow 不做“每个平台一套 UI”，也不做“所有平台完全一样”。

目标是：

```text
结构、品牌、颜色、信息层级统一；
控件交互和系统 chrome 保留平台习惯。
```

也就是说，设置页应该体现 MemoFlow 自己的设计系统，但不强行抹掉 iOS、Android、macOS、Windows 的基础交互习惯。

## 平台分类

平台分类以 form factor 为主，OS 为辅。

```text
phone
  iPhone + Android phone
  统一页面结构、信息层级、颜色语义和设置项位置
  控件细节允许轻微原生化

tablet
  iPad / Android tablet
  独立 tablet 体验
  不是大号手机，也不直接套 desktop

desktop
  macOS + Windows + Linux
  统一桌面端内容宽度、密度、分组和设置结构
  只保留窗口、titlebar、菜单栏、快捷键符号等必要系统差异
```

禁止把 `Apple = iPhone + iPad + macOS` 作为 UI 分类依据。更准确的表达是：

```text
iPhone  = phone + iOS controls
iPad    = tablet + iOS controls
macOS   = desktop + macOS chrome
Windows = desktop + Windows chrome
```

## 设置页骨架

设置页采用“手机端分组感更明显，桌面端更紧凑平面”的方向。

### 手机端

- 分组感更明显。
- 触控友好。
- 行距稍高。
- 分组之间有明确间距。
- 不默认使用大卡片。

### Tablet

- 作为独立 tablet 体验设计。
- 内容宽度收敛，不拉满屏幕。
- 分组仍然明显。
- 后续可以支持侧栏/详情结构。

### 桌面端

- 更紧凑、更平面。
- 内容使用中等宽度，约 720 到 840。
- 少阴影、少大圆角、少装饰。
- 像设置面板，不像手机页面放大。

### 设置首页和二级页

- 设置首页可以使用入口卡片或分区入口，用于承载一级分类和说明。
- 二级设置页默认使用 row / section。
- 只有需要强强调、状态摘要、危险操作时才使用 card。

## 标题和页面 Chrome

标题策略采用中间路线：

```text
phone
  普通顶部标题栏，稳定、省空间，适合大量二级设置页。

tablet / desktop
  可以更像窗口或设置面板标题，但仍应复用 settings/page seam。
```

页面不应在每个 screen 里手写 `Scaffold`、透明 `AppBar`、背景渐变、titlebar 规则或桌面宽度规则。迁移后的页面应通过 `SettingsPage` 或同等语义 seam 表达页面 chrome。

## 颜色语义

采用折中方案：主题色影响交互和选中，固定语义色表达状态。

### 跟随用户主题色

- 主要按钮。
- 选中态。
- 开关打开状态。
- 当前项高亮。
- 链接或可点击强调。
- 小面积 icon highlight。

### 固定语义色

- 危险：error / red。
- 警告：warning / amber。
- 成功：success / green。
- 信息：info / blue 或 theme-compatible info。

### 不建议受主题色影响

- 页面背景。
- 普通卡片背景。
- 普通 section 背景。
- 大面积渐变。
- 普通正文文字。

### 页面级禁止项

迁移后的设置页面不应直接决定：

- 按钮背景色。
- 普通卡片背景。
- 分组背景。
- 开关颜色。
- 危险按钮颜色。

页面应选择语义组件或语义 variant，例如：

```text
SettingsAction.primary
SettingsAction.secondary
SettingsAction.text
SettingsAction.danger
SettingsToggleRow
SettingsStatus.warning
```

而不是在 screen 里直接写颜色、`styleFrom` 或裸 `Switch`。

## 行组件策略

设置页默认以 row / section 为基础，而不是默认以 card 为基础。

常见行类型：

```text
导航行
  标题 + 可选描述/值 + 右箭头
  例：图片床、WebDAV、编辑器工具栏

选择行
  标题 + 当前值 + 下拉/弹窗
  例：语言、字体大小、主题模式

开关行
  标题 + 可选描述 + 开关
  例：震动、提醒、第三方分享

状态行
  标题 + 状态 badge + 可选操作
  例：同步状态、登录状态、权限状态

危险行
  危险语义文字/图标 + 确认
  例：清空缓存、删除本地数据
```

## 带描述的功能开关

已确认采用“一级或重要功能轻量强调，普通开关保持 row”的方向。

```text
重要功能开关
  例如提醒、WebDAV、图片床、图片压缩
  可以使用轻量强调 row 或 SettingsToggleCard 语义
  但不能回到独立卡片视觉系统

普通设置开关
  例如震动、折叠引用、确认退出
  默认使用普通 SettingsToggleRow
```

轻量强调必须克制：

- 不使用强阴影。
- 不使用过大圆角。
- 不使用独立颜色体系。
- 不让 Components 页面看起来像另一套设置系统。

## 功能模块复合按钮

对于 `ComponentsSettingsScreen` 这类功能管理页，重要功能模块不再表达为一整组普通 row，而是表达为“功能模块 + 复合按钮”。

### 结构

每个功能模块显示为一条紧凑复合行，不再拆成独立模块标题层和按钮层：

```text
┌──────────────────────────────────────────────┐
│ 设置与连接  [?]           状态图标 │  开关  │
└──────────────────────────────────────────────┘
  ↑ 左侧：进入详情              ↑        ↑
                              分隔符     右侧：仅开关宽度 + 少量冗余
```

标题和 Tooltip：

- 显示功能模块名称。
- 每个功能标题右侧显示自己的 `?` Tooltip，距离标题约 10px。
- Tooltip 属于该功能标题，不属于页面标题，也不放在整行最右侧。
- Tooltip 内容为中短说明，最多两三行。
- 长说明应进入详情页或帮助页。
- 手机端点击 `?` 显示小浮层。
- 桌面端 hover 和 click 都可以触发小浮层。

按钮层：

- 左侧为进入详情区域，占据除开关外的剩余宽度。
- 右侧为开关区域，只保留开关本身宽度和左右约 5px 冗余，不使用固定 30% 宽度。
- 中间使用分隔符区分两个点击区域。
- 手机端整个右侧区域都可点击切换开关。
- 桌面端只开关和开关附近区域可点击切换。
- 左侧区域用于进入该功能设置详情页。
- 右侧只显示开关，不额外显示“已启用”或“已关闭”文字。

### 视觉

- 手机端圆角：12 到 14。
- 桌面端圆角：8 到 10。
- 背景为 `surface` 加边框。
- hover / pressed 时使用轻微 tint。
- 分隔符在手机和桌面端都显示。
- 状态图标放在左侧区域末端，靠近分隔符。
- 强调程度为中偏轻：能看出是功能模块，但不能像另一套卡片 UI。

### 状态

功能模块状态必须以语义状态表达，颜色只是视觉表现。

```text
未配置
  空心 / 中性色

已配置但关闭
  灰色

错误 / 失败
  红色

权限缺失
  黄色

已启用且正常
  绿色

正在工作
  闪烁绿色
  只用于同步中、上传中、检测中等真实进行中的状态
```

状态图标不应替代开关。开关只表达启用/关闭，状态图标表达配置、权限、错误、健康或工作状态。

## 普通二级设置行

功能模块复合按钮不应扩散到所有设置页。偏好设置、字体、语言、主题、编辑器等二级表单型页面仍默认使用克制的 row / section。

初步方向：

```text
普通选择行
  语言                         中文
  字体大小                     标准
  主题模式                     跟随系统

普通开关行
  震动反馈                     开关
  折叠长内容                   开关
  退出前确认                   开关

危险行
  清空缓存                     红色语义
  重置设置                     红色语义
```

已确认约束：

- 二级表单型设置页默认无图标。
- 设置首页和功能模块可以有图标。
- 开关/功能模块使用短说明或 Tooltip，长说明进入详情页或帮助页。
- 选择项在右侧显示当前值。
- 普通 row 不应承担功能模块复合按钮的左右分区交互。

### 右侧符号

- 进入新页面：轻量空心箭头。
- 打开选择器：当前值 + 下拉符号。
- 开关行：只显示开关。
- 危险行：不显示普通箭头，除非进入确认页或详情页。

### 行高密度

手机端：

- 普通行约 48 到 56。
- 有短描述的行约 64 到 72。

桌面端：

- 普通行约 36 到 44。
- 有短描述的行约 48 到 56。

### Section 标题

复杂页面显示 section 标题，简单页面不显示。

需要标题的情况：

- 页面里有 3 组以上设置。
- 同一页混合不同类型设置，例如外观、编辑器、行为、同步、危险操作。
- 某组设置离开标题后会难以理解。
- 危险操作区。

不需要标题的情况：

- 页面只有一组设置。
- 页面标题已经说明清楚。
- 每一行本身已经非常明确。

危险操作永远独立成 danger section，并带标题。

## 设置首页

设置首页负责导航，不直接承载功能开关。

```text
设置首页
  快速找到设置分类
  显示清楚入口
  不直接塞入功能开关
  不常驻显示普通状态
```

### 入口图标

- 设置首页入口显示图标。
- 图标尺寸统一。
- 默认使用中性色。
- 允许小面积主题色强调。
- 不允许每个入口自己使用一套彩色风格。

### 状态显示

- 首页普通入口默认不显示状态。
- 状态和开关下沉到功能管理页或详情页。
- 只有出现需要用户处理的问题时，才允许显示顶部轻量异常提示。
- 正常情况异常提示隐藏。

可触发顶部提示的例子：

- 同步失败。
- 权限缺失。
- 账号失效。
- 本地数据异常。

### 开关

- 首页不直接放开关。
- 首页只做导航。
- 功能开关放到 `ComponentsSettingsScreen` 等功能管理页。

### 分区组织

- 不强行固定大类。
- 根据现有功能动态组织。
- 同一批入口必须有清楚分组。

## 按钮语义

按钮必须按语义选择 variant，不能由页面直接决定颜色。

```text
主要按钮
  实心
  只用于真正主要动作
  例如保存、完成、连接、启用、继续

次要按钮
  默认边框
  例如取消、稍后、返回

轻量操作
  使用文字按钮
  例如编辑、查看、更多

危险操作
  列表里使用红色文字
  确认弹窗最终动作使用红色实心或高强调
```

按钮尺寸和圆角跟设置页整体平台规则一致：

- 手机端更适合高一点、圆角略大。
- 桌面端更紧凑、圆角略小。

## 选择控件、菜单和弹窗

选择控件分为三类，不应全部做成同一种弹窗。

```text
简单短选项
  下拉菜单 / 小浮层
  例：主题模式、字体大小、行高、启动动作

中等正式选择
  居中弹窗
  例：语言、主题颜色、需要更清楚确认的选择

复杂长列表
  独立页面或大弹窗
  例：系统字体、大量 provider/model、需要搜索或预览的列表
```

### 下拉菜单 / 小浮层

- 从右侧值区域附近弹出。
- 整行点击也可以触发。
- 最小宽度约 180 到 220。
- 最大宽度不超过当前 section 宽度。
- 选中项使用 `✓` 加轻微背景。
- 主题色只用于小面积选中标识。
- 手机端短选项仍可使用小浮层。
- 空间不足时自动调整位置。

### 居中弹窗

- 手机端居中弹窗不用于简单短选项。
- 适用于中等正式选择。
- 宽度根据内容自适应：
  - 短内容约 84% 到 86%。
  - 较长列表约 88% 到 90%。
- 高度自适应并设置最大高度。
- 超出最大高度后内部滚动。
- 弹窗显示短标题。
- 当前选中项使用轻微背景 + check/radio。

### 复杂选择

- 使用独立页面或大弹窗。
- 适用于长列表、搜索、预览、筛选或完整设置流程。

### 确认类弹窗

确认类弹窗按风险分层：

- 普通确认：轻量 dialog。
- 风险、法律、权限确认：更强 dialog，需要明确确认按钮。
- 危险操作确认：强确认 dialog，最终按钮使用危险高强调。

### 浮层 Token

Tooltip、popover、dialog、选择器共用同一套浮层 token。

统一内容包括：

- 背景。
- 边框。
- 阴影。
- 圆角。

区别只在尺寸、位置和内容密度。

## 加载、空状态和错误状态

设置页状态反馈应克制、分层，不应把所有问题都做成全屏 loading 或 toast。

### 加载状态

优先级：

```text
行内加载 > section 内加载 > 整页加载
```

- 行内加载适合单个设置值，例如正在读取系统字体。
- section 内加载适合列表或一组 provider/model。
- 整页加载只在页面核心数据无法显示时使用。

### 空状态

- 使用轻量空状态。
- 不使用大插画。
- 推荐为小图标 + 一句话 + 可选操作按钮。

### 错误状态

错误提示按范围分层：

```text
字段 / 行错误
  行内提示

功能错误
  section 内 error panel

全局错误
  顶部 alert

临时操作结果
  toast / snackbar
```

可恢复错误应优先 inline 表达，不要到处弹 toast。

### 正在工作状态

正在工作不等同于普通 loading。

- 同步中、上传中、检测中等真实工作状态使用功能状态图标的闪烁绿色。
- 普通加载使用 spinner、skeleton 或 loading text。

## ComponentsSettingsScreen 视觉规则

`ComponentsSettingsScreen` 是功能管理列表的首个样板页面。它不应再表现为多个独立标题层级的功能模块，也不应回到旧的卡片墙，而应表现为一个紧凑的功能按钮列表。

### 页面职责

`ComponentsSettingsScreen` 负责管理可选功能组件：

- 提醒。
- 第三方分享。
- 图片床。
- 图片压缩。
- 位置。
- 模板。
- WebDAV 同步。

该页面应回答两个问题：

```text
这个功能是什么？
这个功能现在是否启用，以及是否配置/工作正常？
```

每个功能用一条紧凑复合按钮表达。每个功能标题旁显示自己的 `?` Tooltip，用于解释该功能用途；页面标题旁不放整体 Tooltip。

### 页面结构

```text
功能组件

┌──────────────────────────────────────────────────┐
│ 笔记提醒  [?]              状态图标 │   开关  │
├──────────────────────────────────────────────────┤
│ 第三方分享  [?]            状态图标 │   开关  │
├──────────────────────────────────────────────────┤
│ 图片压缩  [?]              状态图标 │   开关  │
├──────────────────────────────────────────────────┤
│ 位置权限  [?]              状态图标 │   开关  │
├──────────────────────────────────────────────────┤
│ 管理模板  [?]              状态图标 │   开关  │
├──────────────────────────────────────────────────┤
│ 设置与连接（WebDAV 备份） [?] 状态图标 │   开关  │
└──────────────────────────────────────────────────┘
```

页面使用统一列表容器。每一行是一个轻量复合按钮，视觉上共享同一套 surface、边框、圆角、hover/pressed tint、开关和 Tooltip 规则。

### 复合按钮文案

复合按钮左侧文案直接显示用户需要识别和点击的功能入口：

```text
提醒
  笔记提醒

第三方分享
  第三方分享

图片床
  图片床

图片压缩
  图片压缩

位置
  位置权限

模板
  管理模板

WebDAV 同步
  设置与连接（WebDAV 备份）
```

这些文案后续可以根据实际 i18n 词条微调，但原则是列表层级保持单层，不再在按钮外新增每个功能的标题层。

### 复合按钮交互

- 左侧剩余区域点击进入功能详情页。
- 右侧是固定开关区域，只保留开关本身宽度和左右约 5px 冗余。
- 手机端整个右侧区域都可点击切换开关。
- 桌面端只开关和开关附近区域可点击切换。
- 中间分区规则保留，但视觉可以比之前更轻。
- 右侧只显示开关，不显示“已启用 / 已关闭”文字。
- 每个功能标题右侧约 10px 显示自己的 `?` Tooltip。
- `?` Tooltip 解释该功能用途、影响范围、必要权限或配置要求，不放在行尾。
- Tooltip 内容为中短说明，最多两三行。
- 长说明进入详情页或帮助页。

没有详情页的功能模块不应伪装成可进入详情。若某个功能暂时没有详情页，应在实现前决定：

- 为该功能补一个轻量详情/说明页；或
- 左侧区域不可点击，但仍保持单层列表视觉。

### 状态表达

当前版本在每行详情区域末端显示一个语义状态图标，靠近分隔符。状态图标不替代开关：开关只表达启用/关闭，状态图标表达未配置、已配置但关闭、错误/失败、权限缺失、已启用正常或正在工作。

如当前代码没有足够状态来源，首批实现可以使用保守状态，并在后续补齐真实状态推导。

### 视觉密度

- 每行使用中偏轻的按钮外观。
- 行与行之间间距小，形成紧凑列表。
- 列表整体使用边框 + surface。
- 手机端圆角 12 到 14。
- 桌面端圆角 8 到 10。
- hover / pressed 使用轻微 tint。
- 不使用强阴影。
- 不使用大面积主题色背景。
- 不让页面看起来像营销卡片列表。

### 平台差异

手机端：

- 列表分组感更明显。
- 右侧开关区域整块可点击。
- 每个功能标题旁的 `?` Tooltip 点击后显示小浮层。

Tablet：

- 内容宽度收敛。
- 不把手机布局简单拉满。
- Tooltip 点击后显示小浮层。

桌面端：

- 模块更紧凑、更平面。
- 只开关和附近区域切换开关。
- 每个功能标题旁的 Tooltip 支持 hover 和 click。
- 内容宽度遵守设置页中等宽度规则。

## 后续整改判断标准

后续迁移设置页时，应优先判断：

1. 页面是否通过 settings seam 表达页面结构。
2. 颜色是否来自语义 token，而不是 screen 自己挑色。
3. 手机、tablet、desktop 的差异是否来自 platform experience / settings seam。
4. 二级设置页是否默认使用 row / section。
5. 重要功能开关是否只是轻量强调，而不是重新发明卡片系统。
6. macOS 和 Windows 是否共享 desktop experience，只保留必要系统 chrome 差异。
7. 功能管理页是否使用功能模块复合按钮表达“进入详情”和“开关”这两种不同操作。
8. Tooltip 是否位于每个功能标题右侧约 10px，而不是页面标题旁或整行最右侧。
9. 设置首页是否保持导航职责，不直接承载功能开关。
10. 选择器是否按短选项、正式选择、复杂长列表分层，而不是全部使用同一种弹窗。
11. 加载、空状态和错误状态是否按行内、section、全局范围分层。

如果一个页面为了达成视觉效果需要绕过这些规则，应先更新设计记录或 spec，再实现代码。
