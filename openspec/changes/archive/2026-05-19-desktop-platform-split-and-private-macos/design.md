## 背景

公开仓库目前包含共享 Flutter 应用、Android 实现和 Windows 桌面实现。仓库已经在 `memos_flutter_app/lib/private_hooks/active_private_extension_bundle.dart` 预留了私有扩展接入点，`docs/` 下的文档也说明 Apple 平台脚手架和商业运行时有意不出现在公开仓库中。

与此同时，当前桌面行为还没有清晰拆分为“桌面通用”和“Windows 外壳”两层。多个桌面入口、布局组件和窗口行为都明显偏向 Windows。如果缺少治理，新的桌面工作会继续堆积在 Windows 专属路径中，未来 macOS 版本要么重复实现，要么回头拆解大量耦合。

因此，本变更作为多阶段整改的治理总纲：

1. 冻结并记录公开/私有边界。
2. 为所有后续工作引入代码分类模型。
3. 审计并逐步将 Windows 优先桌面行为迁移到共享层或桌面通用层。
4. 创建私有 macOS 覆盖仓库，由它拥有 Apple 原生外壳和商业逻辑。

## 目标 / 非目标

**目标：**
- 定义公开开发加私有 macOS 商业开发的长期仓库策略。
- 定义未来实现工作必须遵守的规范性分层规则。
- 防止商业逻辑泄露到公开仓库。
- 尽量让新的桌面能力落在共享层或桌面通用层，减少重复工作。
- 将 Apple 外壳行为、StoreKit/商业代码和私有发布自动化保留给 macOS 专属归属。
- 将整改拆成多个阶段，让项目可以在清理架构的同时继续交付。

**非目标：**
- 在本变更中实现 macOS 平台。
- 在本变更中执行应用重构或移动代码。
- 定义详细的 App Store 定价、订阅策略或法律政策。
- 保证 Windows 和 macOS 外壳完全 UI 对齐。
- 消除所有平台专属代码；部分外壳和系统集成差异是有意保留的。

## 决策

### 1. 使用公开主仓库加私有覆盖仓库模型

项目必须将公开仓库作为共享 Flutter 业务逻辑和非商业平台代码的事实来源。macOS 商业版必须位于私有仓库，并通过覆盖公开 checkout 的方式集成，而不是成为长期分叉。

理由：
- 完整私有 fork 会让同步成本持续增长。
- 公开仓库已经记录了批准的私有扩展接入点。
- 覆盖式归属可以让共享逻辑从一个地方持续流动。

考虑过的替代方案：
- 整仓私有 fork：拒绝，因为它最大化了分叉和漂移。
- 将所有平台都放进一个私有仓库：拒绝，因为这与保留公开代码、隔离商业 Apple 代码的要求冲突。

### 2. 将所有未来代码分类到四层

所有相关代码必须分类到四个层级之一：

- 共享业务层
- 桌面通用层
- 平台外壳层
- 私有商业层

理由：
- 当前痛点不只是仓库布局，而是架构归属不清。
- 分类模型为评审和未来 change proposal 提供稳定判断依据。

考虑过的替代方案：
- 只按平台目录分类：拒绝，因为很多问题位于共享 Dart 代码内部，而不只在原生平台目录。
- 保持宽泛的“桌面”归属：拒绝，因为这会继续放任 Windows 优先耦合。

### 3. 将 Windows 外壳和 macOS 外壳视为同级，而不是彼此的变体

项目不得把 macOS 建模为 Windows 桌面外壳的小幅换皮。共享行为必须抽取到共享层或桌面通用层，而每个平台外壳必须拥有自己的顶层外观、窗口行为、导航语义和原生集成。

理由：
- macOS 在标题栏、工具栏、侧边栏、菜单、设置窗口和动效气质上与 Windows 有实质差异。
- 复制 Windows 外壳会保留错误抽象。

考虑过的替代方案：
- 复用 Windows 外壳并通过条件样式适配：拒绝，因为这会把结构性平台差异隐藏在脆弱分支后面。

### 4. 将商业代码和 Apple 原生代码保留在私有仓库

私有仓库至少必须拥有：
- `macos/` 平台脚手架
- StoreKit 和收据相关代码
- 权益评估
- 商业运行时分支
- 产品 ID、价格和订阅元数据
- 私有签名和发布自动化
- 有意不公开的 Apple 专属外壳增强

理由：
- 这些区域是主要的保密和分发边界。
- 它们的变化节奏也不同于共享产品逻辑。

考虑过的替代方案：
- 将非敏感 Apple 外壳代码公开：暂缓。未来可以重新评估，但在边界尚未稳定前，初始治理模型将 macOS 版本视为私有优先，以避免意外泄露。

### 5. 整改必须分阶段并面向未来

整改不得要求一次性大重写而阻塞功能开发。实现必须遵循分阶段里程碑：

1. 边界和治理冻结
2. 盘点和分类
3. 桌面通用抽取
4. Windows 外壳收敛
5. 私有 macOS 覆盖仓库启动
6. 私有 macOS 原生外壳和商业运行时推出

理由：
- 项目仍在推进，不能为了大规模重写而暂停。
- 最高杠杆的规则是先停止扩大耦合，再稳定偿还架构债。

## 风险 / 取舍

- [边界过宽] → 私有仓库可能临时拥有比必要范围更多的代码。缓解：桌面通用抽取稳定后重新评审归属。
- [边界过窄] → 商业或 Apple 专属细节可能意外泄露到公开代码。缓解：执行单一接入点规则，并将评审检查绑定到本变更 specs。
- [迁移疲劳] → 开发者可能继续因为短期更快而把新工作直接放进 Windows 专属代码。缓解：要求后续 change 和 PR 评审声明所属层级。
- [桌面通用过度抽象] → 过早抽象可能产生脆弱接口。缓解：只抽取已确认的跨平台行为，而不是假设未来行为。
- [macOS 排期压力] → 私有 macOS 版本可能在部分功能上落后 Windows。缓解：优先抽取共享业务和桌面通用能力，控制功能漂移。

## 迁移计划

1. 批准本总纲变更作为治理架构契约。
2. 将关联任务作为后续实现变更的整改 backlog。
3. 每个后续实现变更都必须声明触及的层级，以及是否修改公开、桌面通用、平台外壳或私有商业代码。
4. 只有在边界和盘点阶段被接受后，才创建私有 macOS 覆盖仓库。
5. 在引入大型私有 macOS UI 表面之前，先将活跃桌面功能推向共享层和桌面通用层。

回滚策略：
- 如果边界证明过于僵硬，可以通过后续治理变更修订文档和 specs。
- 因为本变更引入的是规则而不是运行时行为，回滚风险较低，主要是组织流程层面的调整。

## 待解决问题

- 哪些现有 Windows 向文件应该保留为 Windows 专属，哪些应该提升到桌面通用？
- 边界稳定后，是否应该将某些非商业 macOS 外壳代码移回公开仓库？
- 私有 macOS 版本 v1 需要与 Windows 版本达到多高的视觉和工作流一致性？
- 私有仓库是否保持纯覆盖模型，还是后续将部分私有包移动到专用共享私有包仓库？
