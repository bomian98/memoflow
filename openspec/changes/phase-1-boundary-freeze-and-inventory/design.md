## 背景

总纲变更 `desktop-platform-split-and-private-macos` 定义了整体治理模型，但项目还需要一个更小的操作阶段，把这些规则变成可执行基线。仓库里已经能看出未来整改的大致形状：

- `memos_flutter_app/lib/private_hooks/` 里已经存在公开/私有接入点
- 公开仓库中没有 `memos_flutter_app/macos/` 目录
- 桌面运行时入口集中在 `lib/app.dart`、`lib/main.dart` 和 `lib/application/desktop/`
- `lib/features/home/desktop/` 以及多个直接导入 `WindowsDesktopPageShell` 的功能页面里，外壳/UI 代码明显偏 Windows
- 一些工具代码已经暗示了跨桌面意图，例如同时支持 Windows 和 macOS 的快捷键支持

因此，第 1 阶段需要清楚回答一个问题：当前归属图到底是什么，以及在开始实现变更前必须冻结什么？

## 目标 / 非目标

**目标：**
- 冻结后续工作的已批准公开/私有边界。
- 产出桌面相关区域的第一轮盘点，并为每个区域分配目标层级。
- 识别最优先的 Windows 优先耦合，作为第 2 阶段重构的驱动。
- 创建第 1 阶段完成、允许进入第 2 阶段的验收标准。

**非目标：**
- 在模块之间移动代码。
- 添加 macOS 平台脚手架。
- 引入 StoreKit、权益或发布自动化。
- 在本阶段重命名现有 Windows 向类。
- 一次性解决所有边界模糊项；只要把归属决定明确记录下来，部分项可以先保持 provisional。

## 决策

### 1. 第 1 阶段按区域记录归属，而不是按行做穷尽映射

盘点必须先按目录、文件组和子系统归属展开，而不是试图做完美的逐行分类。

理由：
- 仓库已经足够大，逐行标注会拖慢进度，但不会改善第一阶段决策。
- 下一阶段只需要足够的分辨率来指导重构和仓库分离。

### 2. 初始盘点只使用四种状态

所有被盘点的区域必须被分到以下四种目标状态之一：
- 现在共享
- 桌面通用候选
- Windows 外壳
- 私有 macOS 候选

理由：
- 这些状态和总纲治理模型直接对应。
- 这些状态足够具体，后续变更和代码评审都能直接使用。

### 3. 第 1 阶段必须显式标出优先的 Windows 优先热点

第 1 阶段的设计必须识别至少这些热点：
- `lib/app.dart` 和 `lib/main.dart` 里的桌面 app/runtime 编排
- `lib/application/desktop/` 里的窗口和多窗口协调
- `lib/features/home/desktop/` 里的 Windows 桌面外壳 widget
- 直接依赖 `WindowsDesktopPageShell` 的功能页面
- Windows 专属设置和 Windows 专属 UX 字符串

理由：
- 如果这些区域不被分类，未来最容易强迫 macOS 复制实现。

### 4. 第 1 阶段完成是第 2 阶段实现工作的门槛

在第 1 阶段记录以下内容之前，不应开始大规模的桌面通用抽取或私有 macOS 启动：
- 边界规则
- 归属图
- 热点列表
- 后续重构队列

理由：
- 如果没有这个门槛，项目会回到临时性的 Windows 优先改动。

## 风险 / 取舍

- [盘点太浅] → 后续重构可能发现遗漏耦合。缓解：对不确定区域保留 provisional 备注，并在第 2 阶段继续细化。
- [盘点太重] → 文档工作可能停滞。缓解：把分类保持在子系统/文件组粒度。
- [过早私有归属] → 一些其实可以公开的 Apple 代码可能被过早标私有。缓解：等架构稳定后，再通过后续治理变更放宽边界。
- [阶段混淆] → 开发者可能把第 1 阶段当成可选文档。缓解：把第 1 阶段定位为重构和私有启动的正式前置条件。

## 迁移计划

1. 批准第 1 阶段变更。
2. 产出其 specs 要求的盘点结果。
3. 用这些结果打开第 2 阶段的定向实现变更，推进桌面通用抽取和 Windows 外壳收敛。
4. 只有在边界和盘点结果被接受后，才开始私有 macOS 覆盖规划。
